# CUDA-Image-and-Video-codec (Optimized for Synchrotron Data Processing)

This codec is a high-performance image and video processing framework executed on NVIDIA GPUs using CUDA. Originally developed as a Ph.D. project modeled after the JPEG2000 framework, this specialized adaptation introduces major structural optimizations designed to ingest the massive, high-throughput data streams generated by modern Synchrotron scientific detectors (such as Eiger single-photon counting X-ray detectors).

In scientific experiments, these detectors operate at ultra-high frame rates (often exceeding 1,000 frames per second), creating data streams that scale up to 25 GB/s. Conventional CPU-based compression pipelines (such as LZ4, Bitshuffle, or Zstandard) introduce major computational bottlenecks or fail to deliver viable compression ratios for data storage. By swapping out the traditional sequential EBCOT processing core for a massively parallelized **BPC-PaCo (Bit Plane Coding - Parallel Coding)** engine and optimizing memory management layers, this architecture eliminates conventional CPU stalls and shifts heavy arithmetic loads entirely to the GPU.

---

## 1. Core Structural Adaptations & Optimizations

This branch integrates several optimizations specifically designed to handle single-channel scientific raw data:

* **Linear Padding Optimization:** The baseline codebase implemented padding on 1D representations of multi-dimensional arrays via standard C++ middle-insert vectors (`std::vector::insert`). Because rows must be padded to multiples of 64 to satisfy codeblock boundaries, this caused massive, redundant memory shifts across the entire buffer footprint. This optimization extends the 1D vector linearly by reserving a continuous block of memory directly at the tail end of the image vector, converting an expensive $O(N^2)$ write routine into a rapid linear memory allocation.
* **Removal of YCbCr Color Spaces:** For standard natural image sequences, converting Red/Green/Blue color channels to the YCbCr space minimizes cross-channel spatial dependencies. However, deep grayscale scientific imagery exhibits no such inter-channel correlation. Forcing raw sensor matrices through YCbCr mathematical conversions introduced artificial sharp edges and high-frequency variations, drastically spiking overall entropy. This pipeline bypasses color space transformations to process input vectors directly as independent data fields.
* **Strategic Wavelet Bypass / Copy Kernel:** To eliminate computational overhead, the mathematical calculations for the Discrete Wavelet Transform (DWT) can be bypassed. However, the downstream entropy encoder (BPC-PaCo) depends inherently on an immutable Look-Up Table (LUT) mapped strictly to structural decomposition parameters (Decomposition Level, Sub-band region, and Bitplane position). Completely stripping out the DWT structure caused downstream runtime exceptions. To resolve this, the DWT engine was modified to execute a rapid spatial copy rather than a complex mathematical decomposition, successfully preserving full logical layout compliance while eliminating the computational burden.

## 2. Technical System Requirements

### Software Environment
* **Windows:** CUDA 10+ paired with Microsoft Visual Studio 2019+ (Visual Studio Community 2019 was utilized during core engineering; MSVS 2021 maintains full backward compatibility).
* **Linux:** GCC 8+, CUDA 10+, and CMake 3.10.2+. *Tested natively under Ubuntu 24.04 LTS utilizing the CUDA Toolkit v12.6 framework and NVIDIA Display Driver v580.126.09.*

### Hardware Infrastructure
* **NVIDIA Dedicated GPU Requirement:** Full parallel execution relies strictly on native hardware support. Architectures verified through testing benchmarks include the Pascal (1000 series), Turing (2000 series), and Ampere (3000 series) families, carrying Compute Capabilities of 6.1, 7.5, and 8.6 respectively. Backward compatibility with legacy architectures remains unsupported.

5. Right-click the master project root item, open the **Properties** panel, and navigate to **VC++ Directories -> Include Directories**.    * **GTX 1050, 1060, 1070, 1080 (Pascal):** `compute_61,sm_61`

11. Toggle your target compilation profile to **Release** mode targeting **x64** and run the compiler build task.

### Why configure -xSize 2048 -ySize 2048 -isRGB 1 when the source raw file is a 2527 x 2463 single-channel grayscale image?

#### 1. The Core 8-bit Architecture Constraint
The core internal arrays and execution kernels of this compression engine were natively compiled to parse 8-bit pointer tracks (`unsigned char*`). If you inject a raw 16-bit scientific data layout straight into this 8-bit framework, the system handles each individual 16-bit integer pixel value incorrectly, breaking it apart into two completely distinct 8-bit indices.  

For example, a 16-bit pixel integer holding an absolute value of 1000 (`0x03E8` in big-endian hex formatting) will be parsed as two standalone numbers: a high byte of `0x03` (3) and a low byte of `0xE8` (232).  

#### 2. The RGB Reinterpretation Workaround
To bypass a full re-write of the baseline GPU processing architecture, the codec relies on a structural reinterpretation trick. By passing the parameter `-isRGB 1`, we force the program to handle the data array as a standard 3-channel color image where each composite pixel element spans 3 bytes total (1 byte per channel). This allows the 8-bit codec to seamlessly ingest uncompressed 16-bit raw data without crashing.  

#### 3. Matching the Buffer Footprint Sizing
Because the internal memory representation shifts from a 16-bit single-channel layout (2 bytes per pixel) to an 8-bit 3-channel structure (3 bytes per pixel layout), you must re-calculate and scale down the dimensional values (`-xSize 2048 -ySize 2048`) passed to the program. This step ensures that the total processing byte footprint matches or encapsulates the actual binary payload size of the target raw file:  

* **Original Raw Payload Size:** The raw instrument file measures a resolution of 2527 x 2463 at a bit-depth of 16 bits per pixel (2 bytes per pixel).  
$$\text{Total Original Volume} = 2527 \times 2463 \times 2 \text{ bytes} = 12,442,954 \text{ bytes }(\approx 12.4\text{ MB})$$

* **Reinterpreted Engine Processing Volume:** Setting the operational resolution dimensions to 2048 x 2048 under an RGB multi-channel assumption yields:  
$$\text{Total Engine Grid Allocation} = 2048 \times 2048 \times 3 \text{ channels} = 12,582,912 \text{ bytes }(\approx 12.6\text{ MB})$$

The reinterpreted array layout (12.6 MB) completely covers and encapsulates the original raw detector payload volume (12.4 MB). The extra trailing memory byte allowance acts as a linear padding structure. When decompression is executed, a post-processing crop stage discards the excess trailing padding data to return the file to its original 2527 x 2463 bounds.
